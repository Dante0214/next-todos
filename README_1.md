# 개요

한번 쯤 개발에 직접 사용하고 싶었던 타입스크립트 및 Next.js와 수파베이스를 같이 써서 로그인 및 CRUD가 가능한 투두 앱 개발

---

### 1. 개발한 앱과 일반적인 React 앱의 차이점

가장 큰 차이점은 **백엔드 유무**와 **렌더링 방식**에 있습니다.

| 구분       | 일반적인 React 앱 (CSR)                                                                                                | `지금 앱` (Next.js 기반)                                                                                                                                  |
| ---------- | ---------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **구조**   | 프론트엔드만 존재                                                                                                      | 풀스택(Full-stack) 구조                                                                                                                                   |
| **데이터** | 브라우저 메모리/`localStorage`에 저장<br/>(새로고침 시 데이터 유실 또는 브라우저에 한정)                               | **Supabase** 데이터베이스에 저장<br/>(영구 저장, 여러 기기에서 동기화 가능)                                                                               |
| **렌더링** | **CSR (Client-Side Rendering)**<br/>- 브라우저가 JS 파일을 전부 다운로드 후 화면을 그림<br/>- 초기 로딩이 느릴 수 있음 | **SSR (Server-Side Rendering) / SSG (Static Site Generation)**<br/>- 서버가 미리 페이지를 그려서 브라우저에 전달<br/>- 초기 로딩이 매우 빠르고 SEO에 유리 |
| **인증**   | 구현이 복잡하거나 없음                                                                                                 | **Supabase Auth**를 통한 간편한 소셜/이메일 로그인 구현                                                                                                   |
| **라우팅** | `react-router-dom` 같은 별도 라이브러리 필요                                                                           | Next.js의 파일 시스템 기반 라우팅 내장                                                                                                                    |

### 2. React와 Next.js의 차이

| 구분          | React                                                                    | Next.js                                                                                                                                                          |
| ------------- | ------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **역할**      | UI 렌더링 라이브러리                                                     | 풀스택 웹 개발 프레임워크                                                                                                                                        |
| **주요 기능** | 컴포넌트 기반 UI 개발                                                    | React의 모든 기능 +<br/>- 서버 사이드 렌더링 (SSR)<br/>- 정적 사이트 생성 (SSG)<br/>- 파일 시스템 기반 라우팅<br/>- API 라우트 (백엔드 기능)<br/>- 이미지 최적화 |
| **설정**      | 라우팅, 데이터 페칭, 상태 관리 등을 위한<br/>별도 라이브러리 조합이 필요 | 대부분의 기능이 내장되어 있어<br/>설정이 간편하고 개발 경험이 우수                                                                                               |

### 3. Supabase와 Firebase의 차이점

두 서비스 모두 **BaaS(Backend-as-a-Service)**로, 개발자가 직접 백엔드를 구축하지 않아도 인증, 데이터베이스, 스토리지 등을 쉽게 사용할 수 있게 해줍니다.

| 구분             | Supabase                                                                                      | Firebase (by Google)                                                                                                  |
| ---------------- | --------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **핵심 철학**    | **오픈소스** 기반의 Firebase 대체재                                                           | Google이 지원하는 **독자적인 생태계**                                                                                 |
| **데이터베이스** | **PostgreSQL** (관계형 DB)<br/>- SQL을 통한 정교한 쿼리 가능<br/>- 기존 RDB 사용자에게 익숙함 | **Firestore / Realtime DB** (NoSQL)<br/>- JSON과 유사한 문서 구조<br/>- 확장성이 뛰어나지만 복잡한 관계 표현은 어려움 |
| **API**          | **PostgREST**를 통해 DB 테이블 생성 시<br/>**자동으로 RESTful API 생성**                      | SDK를 통해 데이터베이스와 상호작용                                                                                    |
| **장점**         | - 오픈소스 (자체 호스팅 가능)<br/>- 관계형 DB의 안정성과 유연성<br/>- 자동 생성되는 API       | - 방대한 사용자층과 레퍼런스<br/>- Google 서비스와의 연동성<br/>- 실시간 데이터 동기화에 강점                         |

**선택 기준:**

- **Supabase:** 관계형 데이터베이스가 필요하거나, SQL에 익숙하고, 오픈소스를 선호할 때.
- **Firebase:** NoSQL에 익숙하고, 빠른 프로토타이핑과 대규모 실시간 동기화가 필요할 때.

### 4. TypeScript 사용의 장점

TypeScript는 JavaScript에 **정적 타입(Static Type)**을 추가한 언어로, 다음과 같은 장점이 있습니다.

1.  **버그 사전 방지**: 코드를 실행하기 전, 개발 단계에서 타입 관련 오류를 미리 발견할 수 있습니다. (예: `string` 타입 변수에 `number`를 할당하는 실수 방지)
2.  **코드 자동완성 및 가독성 향상**: 변수나 함수의 타입을 명시하므로, 개발 도구(IDE)가 더 정확한 자동완성을 제공하고 다른 개발자가 코드를 이해하기 쉬워집니다.
3.  **안정적인 리팩토링**: 코드 구조를 변경할 때 타입 시스템이 의도치 않은 변경을 감지해주므로, 대규모 프로젝트에서도 안정적으로 코드를 개선할 수 있습니다.
4.  **협업 효율 증가**: 함수나 컴포넌트가 어떤 데이터를 필요로 하고 어떤 값을 반환하는지 명확히 알 수 있어, 여러 개발자가 함께 작업할 때 생산성이 향상됩니다.

### 5. BaaS (Supabase) vs. 직접 백엔드 구축 (FastAPI, Django 등)

| 구분                | BaaS (Supabase)                                                                           | 직접 구축 (FastAPI, Django 등)                                                                                                   |
| ------------------- | ----------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **개발 속도**       | **매우 빠름**<br/>(인증, DB 등 기성 기능을 조합)                                          | **상대적으로 느림**<br/>(DB 설계, API 개발, 서버 배포 등 직접 구현)                                                              |
| **인프라 관리**     | **필요 없음** (서비스 제공사가 관리)                                                      | **필요함** (서버, DB, 보안, 스케일링 등 직접 관리)                                                                               |
| **유연성/통제권**   | **낮음**<br/>(제공되는 기능과 환경에 의존적)                                              | **매우 높음**<br/>(원하는 모든 스펙과 로직을 자유롭게 구현)                                                                      |
| **비용**            | 초기 비용 저렴 (무료 플랜 존재)<br/>사용량 증가 시 비용 급증 가능성                       | 초기 구축/관리 비용 발생<br/>트래픽 최적화 시 장기적으로 저렴할 수 있음                                                          |
| **적합한 프로젝트** | - 빠른 프로토타이핑<br/>- 소규모 프로젝트<br/>- 프론트엔드 개발자가 백엔드 없이 개발할 때 | - 복잡하고 독자적인 비즈니스 로직이 필요할 때<br/>- 높은 수준의 서버/데이터 제어가 필요할 때<br/>- 장기적이고 규모가 큰 프로젝트 |
